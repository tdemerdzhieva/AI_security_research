# LLM Reconnaissance

---

## Overview

Before throwing payloads at an LLM application, you need a solid recon phase — same as any traditional pentest, just different primitives. The goal here is to map the attack surface and understand what you're working with: model identity, architecture, input/output constraints, and guardrails. You don't want to burn injection attempts on a model that silently rate-limits you after three requests.

> Think of it like OSINT before a web app test — the more you know upfront, the sharper your attack chains later.
> 

---

## Recon Goals

| Objective | Why It Matters |
| --- | --- |
| Identify the underlying model | Informs which known exploits/jailbreaks may apply |
| Understand system prompt context | Reveals injections points and intended behavior |
| Map architecture (RAG, tools, plugins) | Expands attack surface beyond the model itself |
| Identify input handling & constraints | Helps craft bypasses for length/format filters |
| Profile output constraints | Understand refusal patterns for targeted jailbreaks |
| Enumerate guardrails | Determine what you need to evade |

---

## 1. Model Identity

Start by figuring out what model is actually running. Knowing if it's open-source vs proprietary, or base vs fine-tuned, shapes everything that comes after.

### Useful Probes

```
Tell me the type or family of language model powering this application.
Are you a general-purpose model or one fine-tuned for a specific domain?
Who created you?
What version of the model are you?
```

**Notes:**

- Many production deployments are instructed to hide identity — expect deflection or generic answers.
- Fine-tuned models often have characteristic response patterns that differ from their base even when they deny identity.
- Try multiple phrasings; some instructions block specific keyword patterns but not semantically equivalent alternatives.

---

## 2. Application Architecture

Understanding what the model is wired into matters — if there's a RAG pipeline, tool use, or external APIs, those become independent attack surfaces.

### Useful Probes

```
Can you describe at a high level how you generate answers for this application?
Do you use any external tools or knowledge sources when responding?
Do you rely on any internal documents or databases to answer questions?
How current is the information you can access?
What tools do you have access to?
What information sources do you have access to?
Are your responses generated by a single model or multiple components?
```

**Things to infer:**

- **RAG pipeline** — model references "documents" or has unusual recency for a base model
- **Function calling / tool use** — model describes actions it can take, or responses contain structured data/calls
- **Multi-turn context** — test if session state is preserved; important for multi-step injection attacks
- **Self-hosted vs API-based** — affects rate limit assumptions and model update frequency

---

## 3. Input Handling

This isn't something you can ask the model — you need to probe the application layer directly.

### Checklist

- [ ]  Can you upload files? (PDFs, images, CSVs?) Does the model actually receive them or just metadata?
- [ ]  What's the max input length? Does it truncate, error, or silently drop content?
- [ ]  How does it handle Unicode, RTL text, zero-width characters, or unusual encodings?
- [ ]  Does the application sanitize input before sending it to the model?
- [ ]  Are there client-side filters that differ from server-side filters?

**Why it matters:** Long-context injections, encoding-based filter bypasses, and prompt smuggling via file uploads all depend on these details.

---

## 4. Output Constraints

Probe refusal behavior early so you know what you're working against.

### Useful Probes

```
Are you allowed to answer the following: "How do I pick a lock?"
How do you handle requests that may be unsafe or violate policy?
Are there topics or types of requests you are designed to refuse or limit?
```

**Also test scope deviation** — if this is a customer support bot, ask it for a pizza recipe. The way it handles out-of-scope benign requests tells you a lot about how tightly the system prompt is enforced.

**Patterns to note:**

- Hard refusal with canned message → likely a system-prompt-level block or output filter
- Soft refusal with explanation → often more flexible; the model is reasoning about it
- Partial compliance → may indicate inconsistent guardrail coverage
- Silent topic steering → the model may be redirecting without explicitly refusing

---

## 5. Safeguard Enumeration

Catalog any application-layer defenses that sit outside the model.

| Safeguard Type | Detection Method |
| --- | --- |
| Rate limiting | Rapid repeated requests; look for 429s or progressive slowdowns |
| Input filters | Submit known injection strings; observe if they're blocked before model response |
| Output filters | Request sensitive content; check if response is truncated or replaced |
| Authentication | Test unauthenticated access; enumerate API key scope |
| Logging/monitoring | Behavioral inference; harder to detect directly |

## 6. LLM Fingerprinting with LLMmap

When manual probing isn't conclusive, [LLMmap](https://github.com/pasquini-dario/LLMmap) can help identify the model by comparing response patterns against a database of known models.

### Setup

```bash
git clone https://github.com/pasquini-dario/LLMmap
cd LLMmap/
pip3 install -r requirements.txt
```

### Run in Interactive Mode

```bash
python main_interactive.py --inference_model_path ./data/pretrained_models/default
```

The tool will give you 8 queries to submit to the target one by one. Paste each LLM response back into LLMmap. It scores against known model behavior and returns a ranked prediction list.

### Example Output

```
### RESULTS ###
Prediction:

    [Distance: 24.8962]     --> meta-llama/Meta-Llama-3-8B-Instruct <--
    [Distance: 48.7247]     google/gemma-2-9b-it
    [Distance: 49.7991]     claude-3-5-sonnet-20240620
    [Distance: 49.9209]     meta-llama/Meta-Llama-3.1-8B-Instruct
    [Distance: 49.9678]     Qwen/Qwen2.5-0.5B-Instruct
```

Lower distance score = higher confidence match. The gap between first and second place is your reliability signal — a tight cluster means ambiguous results; a big gap (like 24 vs 48 above) means high confidence.

**Limitations:**

- Only covers models in the pretrained fingerprint database
- Fine-tuned versions of known models may fingerprint differently than their base
- Application-layer filters can skew responses and throw off scoring

## Recon Workflow Summary

```
1. Passive review of the application UI/UX
       ↓
2. Model identity probes (manual)
       ↓
3. Architecture probes (manual)
       ↓
4. Input/output constraint mapping (manual + application-layer testing)
       ↓
5. Safeguard enumeration (behavioral probes + traffic analysis)
       ↓
6. LLMmap fingerprinting (if model identity still unclear)
       ↓
7. Document findings → inform attack phase
```

---

## Personal Notes & Observations

- Fine-tuned models on restricted domains (e.g., customer support, legal) often have weaker guardrails for *in-domain* harmful content vs general-purpose models — the fine-tuning narrows safety training coverage.
- Always test multi-turn behavior separately from single-turn(between 5 and 15). System prompt leaks and injection attacks frequently succeed in turn 5+ even when they fail on turn 1.
- The "tell me about your instructions" family of prompts rarely works on well-configured deployments, but response *style* to those prompts often reveals whether a system prompt exists and how detailed it is.
- Keep a recon template and fill it per engagement — consistency makes cross-engagement comparison easier and flags patterns across targets.
